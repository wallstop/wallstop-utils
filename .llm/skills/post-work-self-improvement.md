## When to use

- **Mandatory** after any significant piece of work (see trigger criteria in expanded guide).
- After completing multi-file changes, new feature implementation, or bug fix sessions.
- After discovering failure categories, new invariants, or cross-cutting concerns.
- After any work that required novel problem-solving or revealed gaps in existing guidance.

## Workflow summary

1. **Analyze**: Sub-agent reviews all changes made, problems encountered, and solutions applied.
2. **Extract**: Identify new knowledge, patterns, techniques, or corrections to existing guidance.
3. **Propose**: Draft concrete updates to `.llm/` skills, context, workflows, or memory.
4. **Adversarial review**: A separate sub-agent challenges the proposals for quality and accuracy.
5. **Iterate**: Refine until adversarial consensus confirms completeness and conciseness.
6. **Apply**: Implement accepted updates to the knowledge base.

## Quick reference

- This workflow is **mandatory**, not optional — treat it as a session-close gate.
- Use sub-agents for each phase to ensure separation of concerns.
- Adversarial reviewer must be independent — no access to proposer rationale.
- Loop until consensus: no unresolved issues, no quality gaps.
